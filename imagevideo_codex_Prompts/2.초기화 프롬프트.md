# 최종 프로젝트 생성 프롬프트: Node.js FFmpeg Lyric Video Generator

아래 내용을 기준으로 로컬 Node.js 리릭 비디오 생성 프로젝트를 구현하라. 이 문서 하나만 보고 같은 프로젝트를 만들 수 있어야 한다. 목표는 `input/` 폴더의 가사, 오디오, 선택 배경 이미지를 읽어 실제 재생 가능한 `output/lyric_video.mp4`를 생성하고 ffprobe로 검증하는 것이다.

## 목표

입력:

```text
input/
  lyrics.srt 또는 lyrics.lrc 또는 lyrics.json
  audio.wav 또는 audio.mp3
  background.png 또는 .jpg/.jpeg/.webp 선택
```

출력:

```text
output/production_plan.json
output/subtitles.ass
output/rendered_typography_video.mp4
output/lyric_video.mp4
output/logs/pipeline.log
output/logs/render.log
output/logs/ffmpeg.log
```

최종 MP4는 다음 조건을 만족해야 한다.

```text
video codec: h264
audio codec: aac
pixel format: yuv420p
resolution: 기본 1920x1080
duration: production_plan.json의 project.duration_total과 1.5초 이내 일치
```

## 환경

- Node.js 20 이상
- FFmpeg와 ffprobe가 PATH에 있어야 한다.
- Windows 호환이 중요하다.
- 프로젝트는 ESM을 사용한다. `package.json`에 `"type": "module"`을 둔다.
- 기본 v1 렌더러는 Motion Canvas가 아니라 FFmpeg-only 렌더러다.
- Motion Canvas 관련 파일은 실험용으로 둘 수 있지만 기본 파이프라인을 막으면 안 된다.

## 전체 파이프라인

`src/pipeline/runLyricVideoPipeline.js`는 다음 순서로 실행한다.

```text
1. input 파일 검증
2. npm run plan
3. npm run subtitles
4. npm run render:typography
5. npm run compose -- --audio <audioPath> --no-subtitles
6. ffprobe 기반 최종 검증
```

옵션:

```text
--clean
--debug
--audio <path>
--title <text>
--artist <text>
--skip-plan
--skip-subtitles
--skip-motion
--skip-compose
```

Windows에서 npm script 실행은 `cmd.exe /d /s /c npm run <script>` 방식으로 spawn한다. 각 단계 후 산출물 존재와 0바이트 초과를 확인하고, 실패하면 `output/logs/pipeline.log`에 기록하고 종료한다.

## inputDiscovery.js

`input/` 폴더에서 다음 우선순위로 파일을 찾는다.

```text
lyrics: 정확히 lyrics.json, 그 외 .srt, .lrc, .json 순서
audio: 첫 번째 .mp3 또는 .wav
backgroundImage: 첫 번째 .png, .jpg, .jpeg, .webp
backgroundVideo: 첫 번째 .mp4, .mov, .webm, .avi, .mkv
```

함수:

```js
discoverInputFiles(inputDir = 'input')
isAudioFile(filePath)
isLyricFile(filePath)
isBackgroundImageFile(filePath)
isBackgroundVideoFile(filePath)
```

## lyricParsers.js

`readLyricInput(filePath)`, `parseSrt(raw)`, `parseLrc(raw)`를 구현한다.

규칙:

- JSON은 그대로 parse한다.
- SRT는 블록별 start, end, lyric을 timed segment로 만든다.
- LRC는 각 timestamp를 start로 쓰고 다음 timestamp를 end로 쓴다.
- LRC 마지막 줄은 `+4s`로 끝낸다.
- UTF-8 BOM을 제거한다.
- 빈 줄과 `[Intro]`, `[Verse]`, `[Instrumental]`, `[End]` 같은 메타데이터 전용 줄은 제외한다.
- 잘못된 timestamp, 빈 lyric, end <= start 구간은 제외한다.

반환 예:

```js
{
  lyrics: "가사 전체",
  timed_segments: [
    {
      start: "00:00:04.867",
      end: "00:00:06.702",
      duration: 1.835,
      lyric: "가사 한 줄"
    }
  ],
  source_format: "srt"
}
```

## timecode.js

`formatTimestamp(totalSeconds)`와 `isValidTimestamp(value)`를 구현한다. 형식은 `HH:MM:SS.mmm`이다.

## validation.js

`validatePlannerInput(input)`와 `validateProductionPlan(plan)`을 구현한다.

검증:

- input은 객체
- `lyrics`는 비어 있지 않은 문자열
- `aspect_ratio`는 `16:9`, `9:16`, `1:1`
- `duration_per_line`은 0보다 큰 숫자
- segment id는 1부터 순차적
- segment start/end는 `HH:MM:SS.mmm`

## generateProductionPlan.js

`output/production_plan.json`을 생성한다.

중요 요구사항:

- `discoverInputFiles('input')`로 가사와 오디오를 찾는다.
- `readLyricInput`으로 SRT/LRC/JSON을 모두 처리한다.
- `ffprobe`로 오디오 길이를 읽는다.
- `project.duration_total`은 `max(마지막 가사 end, 오디오 길이)`로 저장한다.
- 오디오가 없거나 ffprobe가 실패하면 마지막 가사 end를 사용한다.

기본값:

```js
aspect_ratio: '16:9'
resolution: '1920x1080'
fps: 30
mood: 'reflective'
genre: 'contemporary pop'
language: 'auto'
visual_style: 'original kinetic typography with soft abstract backgrounds'
```

각 segment는 다음 필드를 가져야 한다.

```text
id, start, end, duration, lyric,
scene_summary, image_prompt, negative_prompt,
motion.camera, motion.background_motion, motion.text_animation, motion.transition,
style.color_palette, style.lighting, style.background, style.emotional_tone,
ffmpeg_notes
```

프롬프트는 저작권 캐릭터, 특정 아티스트/스튜디오 모방, 로고, 워터마크, 노골적/위험한 이미지를 피한다.

## generateAssSubtitles.js

`output/subtitles.ass`를 생성한다.

요구사항:

- `[Script Info]`, `[V4+ Styles]`, `[Events]` 섹션 생성
- 16:9는 1920x1080, 9:16은 1080x1920, 1:1은 1080x1080
- segment가 비어 있으면 오류
- start/end/lyric 검증
- end > start 검증
- 이전 segment와 overlap 금지
- `\`, `{`, `}`, 줄바꿈을 ASS 규칙에 맞게 escape
- `karaoke_highlight`가 true이면 단어별 `{\k...}` 태그를 균등 분배

## renderTypographyVideo.js

`output/rendered_typography_video.mp4`와 `output/logs/render.log`를 생성한다.

CLI:

```text
--plan <path>
--subtitles <path>
--out <path>
--log <path>
--background <path>
--background-video <path>
--no-motion
--motion-strength low|medium|high
```

구현:

- `production_plan.json`에서 aspect ratio, fps, 마지막 segment end를 읽어 렌더 길이를 결정한다.
- 배경 이미지가 있으면 scale/crop 후 `zoompan`으로 느린 움직임을 준다.
- 배경 영상이 있으면 loop, scale, crop한다.
- 배경이 없으면 `nullsrc`와 `geq`로 추상 배경을 만든다.
- readability overlay, vignette, 선택적 noise를 적용한다.
- `subtitles` 필터로 ASS 자막을 burn-in한다.
- 출력은 silent H.264 MP4이다.

출력 옵션:

```text
-c:v libx264
-pix_fmt yuv420p
-movflags +faststart
```

## composeFinalVideo.js

최종 `output/lyric_video.mp4`와 `output/logs/ffmpeg.log`를 생성한다.

CLI:

```text
--audio <path>
--video <path>
--out <path>
--no-subtitles
--title <text>
--artist <text>
```

가장 중요한 안정화 규칙:

- 최종 duration은 `production_plan.json`의 `project.duration_total`을 우선 사용한다.
- plan duration이 없으면 audio duration을 사용한다.
- duration을 알 수 없으면 오류를 던진다.
- 무한 `loop=loop=-1` 영상 스트림에 `-shortest`만 의존하지 말라.
- 최종 영상 스트림은 반드시 유한해야 한다.

필수 필터 구조:

```text
[1:a]atrim=duration=<duration>,asetpts=PTS-STARTPTS,asplit=2[amwav][aout]
[amwav]aformat=channel_layouts=mono[am]
[am]showwaves=s=<width>x80:mode=p2p:colors=0xCCEEFF:rate=<fps>:scale=sqrt[wave]
[0:v]tpad=stop_mode=clone:stop_duration=<duration>,trim=duration=<duration>,setpts=PTS-STARTPTS[vbase]
[vbase]drawbox=... [vbg]
[vbg][wave]overlay=x=0:y=H-80[vwave]
```

이후 선택적으로 subtitle burn, progress bar, title/artist intro overlay를 추가한다.

최종 출력 옵션:

```text
-map [finalVideoLabel]
-map [aout]
-c:v libx264
-pix_fmt yuv420p
-c:a aac
-b:a 192k
-t <duration>
-movflags +faststart
```

이 구조는 FFmpeg가 정상 종료되도록 보장하며, 깨진 MP4의 대표 원인인 `moov atom not found`를 방지한다.

## mediaProbe.js와 validateMediaOutput.js

`ffprobe`로 최종 MP4를 검증한다.

검증:

- 파일 존재
- 0바이트 초과
- video stream 존재
- video codec은 `h264`
- resolution은 plan의 `project.resolution`과 일치
- audio stream 존재
- audio codec은 `aac`
- duration > 0
- plan의 `project.duration_total`과 최종 duration 차이가 1.5초 이하

성공 출력 예:

```text
Media validation passed: 1920x1080, h264/aac, 182.000s, 26247536 bytes
```

## run.bat

```bat
@echo off
chcp 65001 >nul
cd /d %~dp0

echo ========================================
echo  Lyric Video Generator
echo ========================================
echo.

echo [CLEAN] output 폴더 초기화 중...
if exist output (
    rd /s /q output
)
mkdir output
mkdir output\logs
echo [CLEAN] 완료
echo.

echo [1/2] 파이프라인 실행 중...
call npm run lyric-video -- --clean
if %errorlevel% neq 0 (
    echo.
    echo [FAILED] 파이프라인 실패. 로그 확인: output\logs\pipeline.log
    exit /b 1
)

echo.
echo [2/2] 출력 검증 중...
call npm run validate:media
if %errorlevel% neq 0 (
    echo.
    echo [FAILED] 검증 실패.
    exit /b 1
)

echo.
echo ========================================
echo  완료. 출력 파일: output\lyric_video.mp4
echo ========================================
```


## 구현 금지 사항

- 기본 v1 파이프라인에 Motion Canvas 렌더를 의존시키지 말 것
- FFmpeg compose에서 무한 video stream에 `-shortest`만 의존하지 말 것
- 실패한 산출물을 성공으로 처리하지 말 것
- 0바이트 또는 ffprobe 불가 MP4를 성공으로 처리하지 말 것
- 사용자 input 파일을 삭제하지 말 것
- 웹 UI, 렌더 큐, 클라우드 업로드, 계정, 실시간 미리보기를 추가하지 말 것

```bash
npm install
npm run lyric-video -- --clean
npm run validate:media
```

추가 검증:

```bash
node --check src/planner/generateProductionPlan.js
node --check src/subtitles/generateAssSubtitles.js
node --check src/render/renderTypographyVideo.js
node --check src/ffmpeg/composeFinalVideo.js
node --check src/pipeline/runLyricVideoPipeline.js
ffprobe -v error -select_streams v:0 -show_entries stream=codec_name,width,height,duration -of json output/lyric_video.mp4
ffprobe -v error -select_streams a:0 -show_entries stream=codec_name,duration -of json output/lyric_video.mp4
```

정상 예:

```text
video: h264, 1920x1080, 182.000000s
audio: aac, 181.992000s
```

## 핵심 개선 메모

- `production_plan.json`의 `project.duration_total`은 오디오가 가사보다 긴 경우 오디오 길이를 반영한다.
- `rendered_typography_video.mp4`는 가사 마지막 구간까지만 만들어져도 된다.
- `composeFinalVideo.js`가 최종 오디오 길이까지 마지막 프레임을 `tpad=stop_mode=clone`으로 연장한다.
- 마지막 가사 이후 오디오만 남는 구간은 영상 마지막 프레임, waveform, progress bar가 유지된다.
- 최종 duration 기준은 plan duration으로 통일한다.
