# 다곡 검증 레지스트리

## 1. 목적

추가 노래를 분석할 때마다 설계를 곡별 예외 코드로 수정하지 않고, 새로운
데이터 특성을 등록하고 기존 일반 규칙으로 처리 가능한지 판단한다.

## 2. 현재 실제 검증 현황

| 순번 | fixture ID | 패널 수 | 핵심 특성 | 자막 경로 | 설계 판정 |
|---:|---|---:|---|---|---|
| 1 | `fixture_upgrade` | 42 | 긴 연주 구간, Suno 프롬프트 블록, 장기 cue | 강제 정렬 필요 | `HOLD` fixture |
| 2 | `fixture_dessert` | 45 | 섹션 충돌, 장기 cue, 혼합 해상도, 긴 전주 | 강제 정렬 필요 | `REVIEW_REQUIRED` |
| 3 | `fixture_leave` | 30 | 긴 전주·후주, 누락 Outro, 축약 섹션명 | 정규화 우선 | `BUILD_READY` + warnings |

표의 실제 수치는 fixture 검증용이며 제품 로직의 상수가 아니다.

## 3. 세 곡으로 확보된 검증 축

- 30장, 42장, 45장의 가변 패널
- 서로 다른 음원 길이
- 스토리보드 계획 시간이 음원보다 긴 경우와 짧은 경우
- 한글·영문·공백 포함 경로
- 정상 자막과 비정상 장기 cue
- Suno 연주·화음·섹션 지시문
- LRC/SRT 동시 존재
- 긴 전주와 긴 후주
- 누락된 Outro timed tag
- 파일명과 내부 섹션 충돌
- `Verse 1/2`, `Final Chorus` 의미 보존
- 이미지별 해상도 차이
- 모션 phase가 필요한 긴 정지 컷

## 4. 아직 필요한 추가 검증 축

추가 곡은 단순히 개수만 채우기보다 다음 미검증 특성을 우선한다.

- 이미지 50장 이상
- 300초 초과 음원
- LRC만 존재
- SRT만 존재
- 자막이 전혀 없음
- Remix/Extend로 원문과 실제 보컬 순서가 달라진 곡
- Interlude, Hook, Build, Breakdown 중심 구조
- PNG/JPG/WebP 혼합
- 세로 또는 정사각 이미지 혼합
- 이미지 누락·중복처럼 의도적으로 차단돼야 하는 사례

## 5. 추가 곡 등록 절차

1. 원본 해시 기록
2. 자산 inventory 생성
3. 스토리보드와 패널 문서 비교
4. 음원 길이와 이미지 분포 계산
5. LRC/SRT 품질 점수 계산
6. 기존 규칙으로 처리 가능한지 판정
7. 기존 규칙으로 가능하면 fixture만 추가
8. 불가능하면 곡별 예외가 아니라 새로운 일반 규칙 제안
9. 회귀 테스트와 설계 문서를 함께 갱신

## 6. 일반화 승인 질문

새 문제를 발견했을 때 다음 질문을 모두 통과해야 설계를 변경한다.

- 다른 곡에서도 발생 가능한가?
- 곡 제목이나 특정 번호 없이 설명 가능한가?
- 입력 데이터의 증거로 자동 판정 가능한가?
- 설정 정책으로 분리할 수 있는가?
- 기존 정상 곡의 결과를 바꾸지 않는가?
- 회귀 테스트로 재현 가능한가?

하나라도 충족하지 못하면 제품 코드 예외가 아니라 수동 sidecar 또는
검수 항목으로 처리한다.

## 7. 출시 게이트 진행률

```text
실제 분석 완료: 3곡
구현 시작 최소 조건: 충족
MVP 재사용성 검증 목표: 5곡
남은 최소 실제 곡: 2곡
end-to-end 렌더 검증: 미실행
```

분석 곡 수와 end-to-end 통과 곡 수는 별도로 관리한다. 곡을 분석했다고
렌더·CapCut import 검증까지 통과한 것으로 간주하지 않는다.
