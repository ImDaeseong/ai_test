# 주요 결정과 위험

## 1. 아키텍처 결정

### ADR-001: Remotion은 합성 엔진으로 사용

- 결정: Remotion은 생성된 이미지와 영상, 자막, 음악을 합성하는 데 사용한다.
- 이유: 데이터 기반 자동 렌더와 반복 생성에 적합하다.
- 제한: 정지 이미지를 실제 AI 애니메이션으로 생성하지 않는다.
- 대안: FFmpeg 직접 합성, MoviePy, CapCut 수동 편집.

### ADR-002: CapCut은 최종 편집기로 사용

- 결정: MVP는 CapCut 초안 파일을 직접 만들지 않는다.
- 이유: CapCut 내부 초안 포맷과 비공식 도구의 버전 호환 위험을 핵심 경로에서 제거한다.
- 결과: MP4, 섹션별 MP4, SRT, 보고서를 전달한다.
- 향후: pyCapCut은 별도 실험과 호환성 테스트 후 선택 기능으로 추가한다.

### ADR-003: 음악 길이가 최종 기준

- 결정: 스토리보드 권장 시간 합계가 아니라 실제 최종 음원 길이에 맞춘다.
- 이유: 곡마다 길이와 이미지 수가 다르다.
- 결과: 권장 시간은 절대값이 아닌 가중치로 사용한다.

### ADR-004: LRC/SRT는 신뢰 가능한 힌트이지 절대 기준이 아님

- 결정: Remix/Extend 음원에서는 기존 자막을 그대로 신뢰하지 않는다.
- 이유: 전주, 간주, 반복, 생략, 섹션 순서가 달라질 수 있다.
- 결과: 필요 시 최종 음원으로 재정렬하고 신뢰도를 남긴다.

### ADR-005: 저신뢰 자막은 사람 검수

- 결정: 모든 cue를 자동 확정하지 않는다.
- 이유: 노래, 샤우팅, 랩, 코러스 중첩은 일반 음성 정렬보다 어렵다.
- 결과: 자동화 목표는 수동 작업 제거가 아니라 수동 검수 범위 축소다.

### ADR-006: Python과 Remotion 사이를 JSON 계약으로 분리

- 결정: Python이 계획을 만들고 Remotion은 검증된 timeline JSON만 소비한다.
- 이유: 계산과 렌더를 독립 테스트하고 다른 렌더러로 교체 가능하게 한다.

## 2. 주요 위험

| 위험 | 영향 | 가능성 | 대응 |
|---|---|---|---|
| 가창 인식 오류 | 잘못된 자막 | 높음 | 보컬 분리, 원문 강제 정렬, 신뢰도, 사람 검수 |
| Suno 반복/생략 | cue 매핑 오류 | 높음 | 반복·생략 상태 모델과 검수 보고서 |
| 이미지 부족 | 긴 정적 화면 | 중간 | 재사용 모션, 경고, 추가 이미지 요청 |
| 이미지 과다 | 지나치게 빠른 컷 | 중간 | 최소 컷 길이와 제외 정책 |
| Markdown 구조 변경 | 파서 실패 | 중간 | 헤더 검증, schema version, 실제 fixture 회귀 |
| 한글/공백 경로 | 도구 실행 실패 | 중간 | subprocess 배열, Windows 회귀 테스트 |
| ffmpeg/모델 의존성 | 설치 난이도 | 중간 | 진단 명령, CPU 폴백, 선택 기능 분리 |
| GPU 메모리 부족 | 자막 정렬 실패 | 중간 | CPU 옵션, 청크 처리, 사전 진단 |
| Remotion 렌더 시간 | 작업 지연 | 중간 | 저해상도 미리보기, 섹션 렌더, 캐시 |
| CapCut 버전 차이 | 가져오기 문제 | 낮음~중간 | 표준 MP4/SRT 사용, 실제 버전 검수 |
| 원본 덮어쓰기 | 데이터 손실 | 낮지만 치명적 | 읽기 전용 원칙, 별도 workspace, `--force` 제한 |
| 모델/도구 라이선스 | 배포 제한 | 확인 필요 | 구현 전 공식 라이선스 검토 |

## 3. 비용과 성능

- 기본 타임라인과 Remotion 렌더는 로컬 무료 경로를 우선한다.
- 자막 강제 정렬 모델은 다운로드 용량, GPU/CPU 시간, 디스크 사용량을 사전 표시한다.
- 클라우드 렌더나 유료 API는 MVP 기본 경로에 포함하지 않는다.
- 다곡 처리 전 곡당 평균 처리 시간과 임시 저장 공간을 측정한다.
- 비용이 발생하는 기능은 명시적으로 활성화하고 실행 전 예상 비용을 표시한다.

## 4. 확인이 필요한 항목

1. 실제 음악·LRC·SRT가 향후 어느 폴더 규칙으로 저장될지
2. 생성 이미지의 최종 저장 위치와 파일명 규칙
3. SRT와 LRC가 동시에 있을 때 우선순위
4. 자막 표시 기준이 줄 단위인지 단어 단위인지
5. 코러스 반복 자막을 동일하게 반복할지 요약할지
6. 필요한 출력 비율: 16:9만 또는 9:16 포함
7. Remotion과 선택 자막 정렬 도구의 현재 라이선스
8. 실제 사용 중인 CapCut Desktop 버전의 SRT import 동작

이 항목은 구현 전에 샘플 1곡으로 확정하고 설정으로 기록한다.

## 5. 단계별 출시 판단

### Prototype

- 한 곡 timeline 생성
- 30초 Remotion 렌더
- 수동 CapCut import
- 자막 재정렬 없음

### MVP

- 곡별 동적 이미지 수와 음악 길이 처리
- LRC/SRT 기본 사용
- 전체 MP4와 SRT 패키지
- 최소 3곡 검수

### Experimental Alignment

- Remix/Extend 곡 재정렬
- 저신뢰 검수 보고서
- 기본 OFF

### Stable

- 회귀 샘플 확대
- 문서화된 설치와 복구 절차
- 오류와 해결 완료 기록
- 실제 반복 제작에서 품질 확인

## 6. 다음 구체적 단계

1. 공개 가능한 짧은 테스트 오디오 1개와 10장 내외 이미지 fixture를 준비한다.
2. `ProjectManifest`와 `Timeline` JSON schema부터 구현한다.
3. 실제 `01_storyboard.md` 파서와 음악 길이 정규화 테스트를 먼저 통과시킨다.
4. 자막 강제 정렬은 기본 렌더 파이프라인이 검증된 후 별도 단계로 구현한다.
