# 도구 선택 기준 — Claude Code vs Codex

> 복사 위치: 프로젝트 루트 `/TOOL_GUIDE.md`
> 목적: 작업 성격에 따라 어떤 AI 도구를 쓸지 판단하는 기준

---

## 한눈에 보기

| 상황 | 선택 |
|---|---|
| 처음 보는 프로젝트 파악 | Claude Code |
| 여러 파일 동시 분석·설계 | Claude Code |

## Claude Code가 더 나은 경우

### 탐색·분석·설계
- 처음 보는 코드베이스를 빠르게 이해해야 할 때
- 여러 파일 간 의존성, 데이터 흐름을 파악할 때
- 아키텍처 결정이 필요할 때 (패턴 선택, 모듈 분리 등)
- 거버넌스 문서, CLAUDE.md, 체크리스트 작성

### 판단이 필요한 작업
- 보안 검토 ("이 코드가 안전한가?")
- 코드 리뷰 ("이 설계가 적절한가?")
- 버그 원인 분석 (코드 수정 전 이해 단계)
- 복잡한 로직의 테스트 케이스 설계

### CLAUDE.md 자동 로드의 이점이 큰 작업
- 프로젝트 규칙을 자주 참조해야 하는 작업
- 세밀한 컨텍스트 유지가 중요한 장기 작업

---

## Codex가 더 나은 경우

### 실행·반복·배치
- 테스트 실행 → 실패 → 수정 → 재실행 루프
- 파일 여러 개를 같은 패턴으로 수정할 때
- CLI 명령 집중 실행 (`pytest`, `git`, `python` 등)
- 검증 스크립트 연속 실행

### 명확한 범위의 구현
- 설계가 끝난 후 코드 구현 단계
- 이미 이해한 버그의 수정 단계
- 정형화된 파일 생성 (requirements.txt, .env.example 등)
- 테스트 파일 작성 (이미 어떤 함수를 테스트할지 알 때)

---

## 작업 단계별 권장 도구

```
1단계: 이해    → Claude Code  (파일 분석, 현황 파악)
2단계: 설계    → Claude Code  (아키텍처 결정, 계획 수립)
3단계: 구현    → Codex        (코드 작성, 테스트 실행)
4단계: 검증    → Codex        (테스트 반복, 디버그)
5단계: 리뷰    → Claude Code  (코드 리뷰, 보안 점검)
6단계: 문서화  → Claude Code  (CLAUDE.md, README 작성)
```

## 전환 타이밍

### Claude Code → Codex

```
전환 시점:
- 설계·분석이 끝나고 "이제 만들면 된다"는 시점
- Claude Code 토큰 소진 임박
- 반복 테스트 실행이 주 작업이 될 때

절차:
HANDOFF_TEMPLATE.md 작성 → git commit → Codex 부팅 프롬프트
```

### Codex → Claude Code

```
전환 시점:
- 같은 버그를 3회 이상 잡지 못할 때
- 구조적 문제가 의심될 때 (설계 재검토 필요)
- 컨텍스트가 오염되어 판단이 틀리기 시작할 때

절차:
HANDOFF_TEMPLATE.md 작성 → git commit → Claude Code 부팅 프롬프트
```

### 두 도구 모두 막힐 때

```
→ 구조적 문제 가능성 높음
→ TESTING_DONE_CRITERIA.md 6단계 (Feathers 특성화 테스트) 적용
→ 패치 중단, 재설계 검토
```

## 세션 비용 원칙

- **30분 타임박스** — 30분 내 해결 안 되면 태스크를 더 작게 분리
- **짧은 세션 + 명확한 범위** → 두 도구 모두 효율적
- 컨텍스트가 길어질수록 품질 하락 → 일찍 HANDOFF 작성
