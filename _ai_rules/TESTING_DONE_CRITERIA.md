# 테스트 완료 기준 (Done Criteria)

> 복사 위치: 프로젝트 루트 `/TESTING_DONE_CRITERIA.md`
> 작성 시점: **개발 시작 전** — 이 파일 없이 개발을 시작하지 않는다
> 참조 페르소나: Dijkstra, Kent Beck, Dan North, James Bach, Boris Beizer, Michael Bolton, Martin Fowler, Michael Feathers, Lisa Crispin & Janet Gregory, Jeff Sutherland

---

## 먼저 읽어야 할 원칙

### Dijkstra: 목표를 바꿔라

> "테스트는 버그의 존재를 증명할 뿐, 부재를 증명하지 못한다."

**AI 코딩 무한 루프의 근본 원인**: "버그가 없을 때까지"를 목표로 삼기 때문이다.
버그는 항상 더 찾을 수 있다. 목표는 **버그 제로가 아니라 허용 가능한 위험 수준 달성**이다.

### James Bach: Satisfice (충분히 좋으면 충분하다)

> "소프트웨어는 배포의 이점이 남은 위험보다 클 때 충분하다."

완벽한 소프트웨어는 없다. 완료란 **정의된 기준을 통과한 상태**이지, **모든 버그가 사라진 상태**가 아니다.

### Michael Bolton: AI는 테스트가 아니라 확인(Checking)을 한다

> "Checking = 알려진 기준에 대한 자동 검증. Testing = 예상치 못한 것을 탐색하는 행위."

AI는 Checking만 할 수 있다. 진짜 Testing(탐색, 발견)은 사람이 해야 한다.
AI가 "테스트 완료"라고 해도 실제 Testing은 시작도 안 한 상태일 수 있다.

## 0단계: 이 파일을 작성하기 전 답해야 할 질문

```
작업 유형: 신규 프로젝트 / 유지보수 / 신규 기능 (택1)
작업 내용 (한 문장):
담당자:
작성일:
```

**Kent Beck 원칙 적용**: 아래 완료 기준을 작성하기 전에는 AI에게 코딩을 시키지 않는다.
완료 기준 = AI에게 주는 명세다. 명세 없이 AI를 돌리면 AI도 어디서 멈춰야 할지 모른다.

## 1단계: 인수 조건 정의 (Dan North — BDD 방식)

> 모호한 조건이 무한 루프를 만든다. Given-When-Then으로 구체화하면 AI도 멈출 수 있다.

**나쁜 조건**: "로그인이 잘 된다"
**좋은 조건**:
```
Given: 등록된 사용자가 올바른 이메일과 비밀번호를 입력했을 때
When:  로그인 버튼을 누르면
Then:  대시보드 화면으로 이동하고 세션이 생성된다
```

### 이 작업의 인수 조건

```
조건 1:
  Given:
  When:
  Then:

조건 2:
  Given:
  When:
  Then:

조건 3:
  Given:
  When:
  Then:
```

**완료 기준**: 위 조건이 모두 통과될 때 이 작업은 끝난다. 추가 수정은 다음 작업으로 분리한다.

## 2단계: 테스트 전략 계획 (Martin Fowler — 테스트 피라미드)

> AI 코딩은 기본적으로 테스트 피라미드를 역삼각형으로 만든다. 의도적으로 피라미드를 세워야 한다.

```
올바른 피라미드          AI 코딩 기본값 (역삼각형)
    /\                        ____
   /E2E\          vs         /E2E \
  /------\                  /------\
 /통합테스트\               / 통합  \
/------------\             /--------\
/ 단위 테스트  \           /  단위?  \
```

| 테스트 유형 | 비율 목표 | AI 사용 가능 | 이 작업 계획 |
|---|---|---|---|
| 단위 테스트 (함수/모듈 단위) | 70% | 가능 | |
| 통합 테스트 (모듈 간 연동) | 20% | 부분 가능 | |
| E2E 테스트 (전체 흐름) | 10% | Mock 위험 있음 | |

## 3단계: 테스트 사분면 계획 (Lisa Crispin & Janet Gregory)

> AI는 Q1과 Q2만 한다. Q3과 Q4는 사람이 반드시 별도로 계획해야 한다.

```
                비즈니스 관점
                      |
    Q2 기능 테스트     |     Q3 탐색적 테스트
    (인수 조건 확인)   |     (사용자 경험, 예상 밖 동작)
    AI 가능           |     사람 필수
    ─────────────────────────────────────────
    Q1 단위/컴포넌트   |     Q4 성능/보안/부하
    테스트             |     테스트
    AI 가능           |     사람 필수
                      |
                기술 관점
```

**이 작업에서 Q3, Q4 계획:**
```
Q3 탐색적 테스트 담당자:
Q3 탐색 시나리오:
  1.
  2.

Q4 성능/보안 테스트 계획:
  성능 기준:
  보안 확인 항목:
```

## 4단계: 버그 심각도 분류 및 완료 기준 (Jeff Sutherland — DoD)

> DoD는 개발 완료 후 만드는 것이 아니라 시작 전 팀이 합의하는 계약이다.
> 개인 프로젝트라도 본인과의 계약이다. 중간에 기준을 낮추지 않는다.

### 버그 심각도 분류

| 등급 | 기준 | 배포 가능 여부 |
|---|---|---|
| **P0 (치명)** | 서비스 중단, 데이터 손실, 보안 취약점, 결제 오류 | 반드시 수정 후 배포 |
| **P1 (심각)** | 핵심 기능 오작동, 인수 조건 미통과 | 반드시 수정 후 배포 |
| **P2 (보통)** | 부기능 오작동, 우회 가능한 불편 | 문서화 후 배포 가능 |
| **P3 (낮음)** | UI 깨짐, 오탈자, 사소한 불편 | 다음 배포에 포함 |

### 작업 유형별 DoD

#### 신규 프로젝트

```
DoD (이것이 모두 충족되어야 완료다):
  [ ] 1단계 인수 조건이 모두 Given-When-Then 기준으로 통과된다
  [ ] P0, P1 버그 0개
  [ ] 테스트 피라미드 계획대로 단위/통합 테스트가 존재한다
  [ ] Q3 탐색적 테스트를 사람이 직접 실행했다
  [ ] Q4 보안 체크리스트(AI_CODING_REVIEW.md)를 사람이 직접 통과시켰다
  [ ] 로그가 구조화된 형식으로 출력된다
  [ ] 알려진 P2, P3 이슈가 목록화됨

알려진 미해결 이슈:
  이슈 | 심각도 | 처리 예정일
```

#### 유지보수 (버그 수정)

```
수정 대상 버그:
재현 조건 (Given-When-Then):
  Given:
  When:
  Then (현재 잘못된 동작):
  Then (기대하는 올바른 동작):

DoD:
  [ ] 재현 조건에서 기대하는 동작이 확인된다
  [ ] 수정 전 동작하던 기능이 수정 후에도 동작한다 (회귀 목록 아래 작성)
  [ ] 수정으로 인한 새로운 P0, P1이 없다
  [ ] Michael Feathers 특성화 테스트: 수정 전 동작을 테스트로 문서화했다

회귀 확인 대상:
  기능 1:
  기능 2:
```

#### 신규 기능 추가

```
기능 설명:

DoD:
  [ ] 1단계 인수 조건이 모두 통과된다
  [ ] 기존 기능 회귀가 없다 (아래 목록 확인)
  [ ] P0, P1 버그 0개
  [ ] 엣지 케이스가 정의되고 결과가 기록된다
  [ ] Q3: 사람이 예상치 못한 사용 방식으로 탐색 테스트를 실행했다

기존 기능 회귀 확인 목록:
  기능 1:
  기능 2:

엣지 케이스:
  케이스 | 예상 결과 | 실제 결과 | 심각도
```

## 5단계: 무한 루프 원인 진단 (Boris Beizer — 농약 역설)

> AI 무한 루프의 진짜 원인: 같은 방식으로 같은 테스트를 반복하면 새로운 버그를 찾을 수 없다.
> 벌레가 농약에 내성이 생기듯, 코드도 반복 테스트에 "내성"이 생긴다.

### 무한 루프 판단 기준

```
현재 수정-발견 사이클 횟수: ___
같은 모듈에서 발생한 사이클: ___
예상 대비 소요 시간 초과: ___ 배
```

**농약 역설 신호 (3개 이상 = 테스트 전략 자체를 바꿔야 한다):**
- [ ] AI가 같은 위치를 3회 이상 수정했다
- [ ] 수정 후 다른 위치에서 새 버그가 계속 나온다
- [ ] AI가 같은 버그에 대해 매번 다른 원인을 설명한다
- [ ] 고칠수록 코드가 복잡해지고 있다
- [ ] 이전 수정을 되돌리는 상황이 발생했다

**농약 역설 탈출 방법:**
```
현재 AI 테스트 방식을 중단하고, 다른 테스트 전략으로 전환한다.

전환 옵션:
1. 다른 각도에서 Q3 탐색적 테스트 실행 (사람이 직접)
2. 단위 테스트 레벨로 내려가서 해당 함수만 집중 검증
3. 구조적 문제 여부 판단 (아래 기준 적용)
```

## 6단계: 구조적 문제 판단 (Michael Feathers — 특성화 테스트)

> 패치로 해결되지 않는 문제는 구조적 문제다. 이때 Feathers의 특성화 테스트가 탈출구다.

### 구조적 문제 판단 체크리스트

- [ ] 같은 모듈에서 수정 사이클이 3회 이상 반복됐다
- [ ] 수정 범위가 최초 예상보다 2배 이상 넓어졌다
- [ ] AI가 원인에 대해 일관된 설명을 못 한다
- [ ] 모듈 간 의존성이 예상보다 깊게 얽혀 있다

**3개 이상 체크 = 구조적 문제, 패치 중단**

### 구조적 문제 시 Feathers 특성화 테스트 절차

```
1. 수정 작업을 완전히 중단한다

2. 현재 코드가 실제로 하는 동작을 테스트로 기록한다 (이것이 특성화 테스트)
   → "올바른 동작"이 아니라 "현재 동작"을 기록하는 것이 목적
   → 이 테스트가 리팩터링 중 의도치 않은 변경을 감지하는 안전망이 된다

3. 안전망이 생긴 뒤에 구조 개선을 시작한다

재설계 범위:
재설계 방향:
기존 코드 재사용 여부: 전체 폐기 / 부분 재사용
```

## 7단계: AI가 할 수 없는 테스트 — 인간 검증 필수 항목

> Michael Bolton: AI는 Checking(검증)만 한다. Testing(탐색)은 사람의 영역이다.
> 이 항목을 AI가 "완료"라고 해도 테스트가 끝난 것이 아니다.

| 항목 | Bolton 관점 | 확인 방법 |
|---|---|---|
| 실제 비즈니스 규칙 | AI는 도메인 지식이 없다. 기술적으로 통과해도 업무적으로 틀릴 수 있다 | 기획자/담당자 직접 확인 |
| 탐색적 테스트 (Q3) | AI는 시나리오 밖을 탐색하지 않는다 | 사람이 "예상 밖으로" 사용해본다 |
| 사용자 경험 | AI는 UX를 감각으로 판단하지 못한다 | 실제 화면 직접 조작 |
| 실제 데이터 경계값 | AI는 Mock 데이터로 테스트한다. 실제 데이터와 다르다 | 프로덕션 유사 데이터로 테스트 |
| 보안 취약점 (Q4) | AI는 공격자 관점이 없다 | AI_CODING_REVIEW.md 사람이 직접 확인 |
| 성능 기준 (Q4) | AI는 실제 부하를 시뮬레이션하지 않는다 | 실제 트래픽 수준 테스트 |
| 외부 연동 실제 동작 | AI는 Mock으로 통과시킨다. 실제 API는 다르다 | 실제 API/DB 연동 확인 |

**이 작업에서 사람이 직접 확인할 항목:**
```
확인 항목:
확인 담당자:
확인 완료 여부:
```

```
         |                     |                   |
         |                     |                   |
```

## 9단계: James Bach — 중단 발견법 (언제 멈추는가)

> "언제까지 테스트해야 하나?"의 실전 답변.
> 아래 조건 중 하나라도 충족되면 테스트를 종료할 수 있다.

| 중단 조건 | 확인 |
|---|---|
| **DoD 달성**: 4단계의 완료 기준이 모두 통과됨 | [ ] |
| **신뢰 달성**: 고위험 영역이 모두 검증됨 | [ ] |
| **시간 소진**: 정해진 테스트 시간이 완료됨 | [ ] |
| **위험 수용**: 남은 버그가 P2 이하이고 문서화됨 | [ ] |
| **수익률 하락**: 추가 테스트가 새 버그를 찾지 못하는 상태 3회 연속 | [ ] |

**중요**: 위 조건을 충족했는데도 계속 테스트하는 것은 낭비다.
더 중요한 것은 남은 위험을 문서화하고 의식적으로 수용하는 것이다.

## AI 테스트 프롬프트 원칙 (Kent Beck + Dan North 기반)

```
나쁜 프롬프트 (무한 루프 유발):
"이 코드를 테스트하고 오류를 모두 고쳐줘"
→ 완료 기준이 없어서 AI가 완벽을 추구하며 영원히 돌아감

좋은 프롬프트 (Kent Beck TDD 방식):
"아래 인수 조건을 Given-When-Then 형식으로 테스트해줘.
 [인수 조건 붙여넣기]
 P0, P1만 수정해줘.
 P2 이하는 목록만 만들어줘.
 같은 위치에서 3회 이상 버그가 나오면 수정 중단하고
 '구조적 문제 의심'으로 보고해줘.
 테스트가 완료되면 중단 조건(DoD 달성 여부)을 명시해줘."
```
